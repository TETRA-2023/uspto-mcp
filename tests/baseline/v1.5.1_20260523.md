```

Input: (none)
```

```json

  "reachable": true,
  "status_code": 404,
  "url": "https://ppubs.uspto.gov"
```

**Note:** HTTP 404 on root is expected — PPUBS is a SPA; the root path has no static
resource. The API endpoints under `/api/` and `/pubwebapp/` return 200. Reachable = true
is the correct signal.

## S02 — OR-flood: NL query vs BRS-AND equivalent (T03/T04/T06/T07/T10 golden scenarios)

Measures the inflation factor introduced by PPUBS defaulting to OR between space-separated
terms. These are **count-only** calls (`ppubs_get_search_count`).

| Topic | NL query (OR default) | BRS-AND equivalent | OR:AND ratio |
|---|---|---|---|
| Graphene batteries | `graphene battery` → **2,458,462** | `graphene AND battery` → **53,254** | **46×** |
| AI agent prosecution | `AI agent patent prosecution` → **14,929,952** | `AI AND agent AND patent AND prosecution` → **1,346** | **11,093×** |
| CRISPR gene editing | `CRISPR gene editing cancer treatment` → **4,485,791** | `CRISPR AND gene AND editing AND cancer AND treatment` → **19,342** | **232×** |
| Solid-state batteries | `solid state lithium battery electric vehicle` → **17,294,722** | `solid AND state AND lithium AND battery AND electric AND vehicle` → **65,040** | **266×** |

**Key finding:** Without BRS-AND rewriting, an LLM using NL queries gets result sets
46–11,093× too large. The `ppubs_get_search_count` endpoint correctly reflects this
discrepancy. See also S04 (search result relevance comparison).

## S03 — Zero-result + query_hint (T08 golden scenario)

```
Tool: ppubs_get_search_count
Input: query='"artificial intelligence patent prosecution agent"'
```

```json
{
  "total": 0,
  "query": "\"artificial intelligence patent prosecution agent\"",
  "sources": ["US-PGPUB", "USPAT", "USOCR"],
  "query_hint": "Zero results — if the query used quoted phrases (e.g. \"AI Agent\"), retry with bare AND terms (e.g. AI AND Agent)."
}
```

**Current behavior:** Returns 0 + `query_hint` string. The LLM must read `query_hint`
and issue a second call. **2 tool calls** required to resolve this scenario.

**Post-Tier-1 target:** Auto-retry internally, resolve in **1 tool call** with
`auto_retry: { query_used: "artificial AND intelligence AND patent AND prosecution AND agent", total: <N>, ... }`.

```json

  "total": 3,
  "num_found": 3,
  "page": 0,
  "per_page": 3,
  "total_pages": 0,
  "results": [
    { "inventionTitle": "SYSTEMS AND METHODS FOR ALIGNMENT OF OPTICAL COMPONENTS", "applicantName": ["Meta Platforms Technologies, LLC"], "score": 4.65 },
    { "inventionTitle": "HETEROSTRUCTURES WITH SCANDATE METAL OXIDE AND POTASSIUM TANTALATE LAYERS", "applicantName": ["Wisconsin Alumni Research Foundation"], "score": 2.51 },
    { "inventionTitle": "METHODS, ARCHITECTURES, APPARATUSES AND SYSTEMS FOR USER PLANE TRAFFIC GATING BASED ON ENERGY COST", "applicantName": ["InterDigital Patent Holdings, Inc."], "score": 3.62 }
```

**Relevance assessment:** ZERO results topically relevant to graphene or batteries.
OR semantics match any patent containing either word — the OR flood visible in S02 is
confirmed in the result quality.

### S04b — BRS-AND query: `graphene AND battery`

```json
{
  "total": 3,
  "num_found": 3,
  "page": 0,
  "per_page": 3,
  "total_pages": 0,
  "results": [
    { "inventionTitle": "HYBRID SUPPORT FOR METALLOCENE CATALYST", "applicantName": ["KING FAHD UNIVERSITY OF PETROLEUM AND MINERALS"], "score": 4.76 },
    { "inventionTitle": "MEMBRANE ELECTRODE ASSEMBLY, ELECTROLYSIS CELL, AND ELECTROLYSIS DEVICE", "applicantName": ["KABUSHIKI KAISHA TOSHIBA"], "score": 6.21 },
    { "inventionTitle": "ELECTRODE AND RECHARGEABLE LITHIUM BATTERY INCLUDING THE SAME", "applicantName": ["SAMSUNG SDI CO., LTD."], "score": 10.05 }
  ]
}
```

**Relevance assessment:** All 3 results are in the electrochemistry / energy storage
domain — visibly more relevant than S04a. Confirms AND rewriting produces qualitatively
better results even before semantic re-ranking.

### Pagination observation

Both S04a and S04b returned `total: 3` with `limit=3`. The `total` field maps to
`totalResults` in the PPUBS search response, which reflects the **family-filtered**
result window — not the raw corpus count. Family deduplication (`docFamilyFiltering:
"familyIdFiltering"`) collapses many patents into representative records.

**Implication:** `ppubs_search_patents.total` is NOT comparable to
`ppubs_get_search_count.total`. For reliable corpus counts, always use
`ppubs_get_search_count`. This discrepancy is not documented in the current tool
descriptions and should be clarified in Tier 1 description updates.

## S07 — T09 multi-step pattern: search + 3× detail fetch

```
Workflow: "Find the 3 most recent patents about transformer attention mechanisms"

Call 1: ppubs_search_patents(query="transformer AND attention AND mechanism", limit=3)
  → 3 results returned
Call 2: ppubs_get_patent_by_number("20260141076")  ─┐
Call 3: ppubs_get_patent_by_number("20260141013")   ├─ parallel capable
Call 4: ppubs_get_patent_by_number("20260141513")  ─┘
```

**Total MCP tool calls:** 4  
**Total internal PPUBS API calls:** 7 (1 search + 3×[1 BRS lookup + 1 highlight fetch])  
**Total wall time (estimated):** ~3–5 seconds (3 detail calls sequential in current impl)

**Post-Tier-2 target:** 1 `search_patents_with_details` call with internal
`asyncio.gather` on detail fetches → same result in 1 MCP tool call.

**Results returned:**
- `20260141076` — "Detection of Software Vulnerabilities Utilizing Parallel Large-Language Models" (Saudi Arabian Oil, 2026-05-21)
- `20260141013` — "Real-Time Search Engine URL Generation and Content Discovery" (Microsoft, 2026-05-21)
- `20260141513` — "High Resolution Medical Image Processing for Autonomous Diagnostics Using Vision Transformer and Machine Learning" (Digital Diagnostics Inc., 2026-05-21)

All 3 contain "transformer" in text but are not about attention mechanisms per se.
Reflects PPUBS keyword-match (not semantic) retrieval.

## Observations for Tier 1 description updates (not in current tool docs)

1. `ppubs_search_patents.total` ≠ `ppubs_get_search_count.total` — family filtering
   causes the search endpoint to return a collapsed family count, not the raw corpus
   count. Use the count tool for reliable totals.

2. PGPUB vs USPAT field population differences — `numberOfClaims/Drawings/Figures`,
   `primaryExaminer`, `legalFirmName`, `assigneeName` are null on all PGPUB records.
   `applicantName` is the PGPUB substitute for `assigneeName`.

3. `check_ppubs_status` returning HTTP 404 is not an error — document this to prevent
   false-alarm triage.

4. `inventionTitle` in search results may contain `<span>` highlight HTML when the
   query term appears in the title (see S04b result 3). Downstream consumers parsing
   titles should strip HTML.
